P1 · Consumer model · agent-first

Nuri is optimised for an agent reader. Three agent personas plus an operator are the consumer set; humans browsing cold are not a primary use case.

Personas
spec-authoring composing migration operator
Reason
Figma is the wrong tool in an agent-first workflow — agents can't iterate on Figma. The iteration loop is operator-prompts → agent-composes → translate. No design-handoff step. README is the "what is Nuri" surface for newcomers; the docs site is a catalogue.
Applies to
every Nuri surface (component pages, foundation pages, principles, AGENTS.md)

decisionlog.md decision 21 · RISKS.md R6 (personas framing acknowledged as untested)

P2 · 2-layer token architecture

A flat primitive catalogue (theme-agnostic) plus a semantic layer that composes mode × accent. No Layer 1 swap, no Layer 3 at the system level.

Primitive
tokens-primitive.css flat catalogue · every colour exists as -light and -dark · never reactive to data-theme
Semantic
tokens-semantic.css role-based tokens · composes mode × accent · components consume these
Reason
Theme decisions are isolated from raw values; maps 1:1 onto DTCG mode-aware references in the RN sync pipeline. An earlier draft used theme-swapped primitives (--nuri-color-neutral-1 meant a different hex in each theme); it hid the "which value plays which role" decision inside the primitive layer.
See
decisionlog.md decision 1 · Colour · Primitive · Colour · Semantic

P3 · Components consume semantic, never primitive

Component CSS references --nuri-bg-*, --nuri-text-*, --nuri-border-*, --nuri-accent-*. Never --nuri-color-*. The only place hex literals live is tokens-primitive.css.

Allowed prefixes
--nuri-bg-* --nuri-text-* --nuri-border-* --nuri-accent-* --nuri-focus-ring --nuri-<component>-*
Forbidden in component CSS
--nuri-color-* primitive references in component code · hex literals anywhere
Reason
Indirection makes the component's intent visible (it asks for a role, not a hex) and lets the semantic layer evolve without touching components.
Applies to
lib/components/*/<name>.css
See
Button · button.css

P4 · Mode × accent composition · asymmetry by family

Two attributes on <html> drive every theme decision: data-theme (light|dark) and data-accent (neutral|lilac|…). The semantic layer declares per accent how it reacts to mode. The reaction is asymmetric between accent families.

Saturated family
gray blue red jade full mode swap · accent-solid inverts in dark
Bright family
lilac amber mint yellow sky brand frozen · accent-solid + accent-on-solid identical in both themes · only accent-fg + accent-bg-subtle adapt
Reason
Saturated scales support mode symmetry (light/dark are equivalent). Bright scales carry brand identity that must remain recognisable across themes. The asymmetry is declared per accent in the semantic cascade so it's self-documenting.
See
AGENTS.md · Cascade ordering · Colour · Semantic (live cascade) · tokens-semantic.css · decision 33 (Format B inline docs · implementation surface)

P5 · 3-tier component model

Components fall into three tiers by how they interact with data-accent. All three use the same underlying mechanism (CSS cascade via the attribute); the tier names describe intent.

Tier 1 · Scope-providing
Card Modal Sheet accent prop mirrored to data-accent on the root · subtree of children inherits
Tier 2 · Accent-consuming leaf
Button Tag Chip IconButton accent prop mirrored to data-accent on the inner element · only the component's own resolution is affected
Tier 3 · Generic scope primitive
<nuri-scope> no visual · display: contents · accepts mode / accent / neutral / font / density · web-only (RN uses React Context analogues)

decisionlog.md decision 6 · Button (Tier 2 example)

P6 · Mobile-first pressed-only interaction

RN has no :hover. Nuri components have :active only. Semantic tokens use -pressed, never -hover. The docs chrome may use :hover because it's docs UI, not a Nuri component.

Allowed pseudo-classes in component CSS
:active :focus-visible :disabled
Forbidden in component CSS
:hover no -hover suffix tokens in semantic either
Token suffix
--nuri-accent-solid-pressed · --nuri-bg-pressed
Reason
The canonical consumer of Nuri tokens is React Native, not the docs site. RN's only interactive state beyond default is press (onPressIn / onPressOut). Token names reflect the RN consumer's reality.
See
decisionlog.md decision 3 · RISKS.md R1 (touch behavioural parity)

P7 · Variant naming · solid / soft

Variant describes visual treatment, not hierarchy. solid = filled with active accent; soft = chrome-only. Default is soft (the quieter alternative). Composes with accent: variant chooses how, accent chooses which colour.

Variants
solid soft (default)
solid · semantics
filled with the active accent · accent-driven · for CTAs
soft · semantics
tinted with chrome tokens · accent-invariant · the quieter default
Reason
"primary / secondary" implies hierarchy and doesn't generalise to rails-style UI where buttons are pari-level but different colours. Aligns with Radix Themes vocabulary.
See
decisionlog.md decision 5 · Button

P8 · Unit rule · rem for type, px for everything else

Only font-size is in rem — it scales with the user's text-size preference. Spacing, radii, border widths, shadow offsets stay in px. Line-heights are unitless ratios (decision 29).

rem
font-size
px
spacing radii border widths shadow offsets
unitless
line-height ratios inside type scale mirrors RN: lineHeight = size × ratio at render
Reason
Faithful preview of RN's allowFontScaling: type grows with user preference, containers stay put, overflow / wrap behaviour is visible. Containers in px so layout doesn't drift when the user scales text.
See
Typography

P9 · DTCG type inference via naming convention

Every token's prefix encodes its DTCG $type. The pipeline emits correct types without metadata files. Semantic tokens are references and inherit type from the primitive they reference.

color
--nuri-color-*
dimension
--nuri-px-* · --nuri-space-* · --nuri-size-* · --nuri-radius-* · --nuri-font-size-* · --nuri-border-*
fontFamily
--nuri-font-family-*
fontWeight
--nuri-font-weight-*
themed scales
--nuri-color-<scale>-<step>-{light,dark} — e.g. --nuri-color-gray-1-light. The 4-deep name encodes scale + step + theme variant; the DTCG path nests one level per segment.
duration
--nuri-duration-*
Reason
Pipeline-friendly: parser infers DTCG $type from the prefix. No metadata files to keep in sync. New token types require extending this table (and the parser).
See
lib/docs/tokens.js (browser parser) · pipeline/tokens-parser.js (Node parser, thin slice) · RISKS.md R2 (pipeline schema risk)

P10 · Tokens become RN at build, not runtime (Pattern C)

Source of truth: CSS vars in /styles. Pipeline: CSS → DTCG JSON → custom emitter (pipeline/parsers/semantic.js) → typed RN tokens. Style Dictionary is bypassed for the RN-only target — the cascade-aware emitter is terminal (decision 2.1 · 7.1); SD re-enters only if a second target platform lands (iOS / Android / Figma sync). The emitter resolves every reference at build, so RN ends up with literal values + typed TokenPath strings, not var() chains.

Source
tokens-primitive.css tokens-semantic.css
Intermediate
DTCG tokens.json W3C standard · machine-readable spec
Output
custom emitter → typed RN build/tokens.ts references resolved at build · literal values + TokenPath strings · per-component numerics in build/components/<name>.ts (one file per component, not a .spec.json) · decision 34
Web side
CSS vars consumed directly · references resolve at runtime via cascade · build-free prototyping (can run as a Claude chat artifact)
Divergence accepted
Web prototypes can use patterns RN cannot (deep var() chains, runtime token mutation, reference cycles that no-op on web but fail at build resolution). A future lint will flag the divergent patterns before they reach migration.
See
decisionlog.md decision 2 · RISKS.md R2 · R3 · R4

P11 · Parsimony at the spec layer · ships what's consumed

Nuri's primitive layer — and downstream surfaces (components, skills, pages) — ships only what has a current consumer at ship time. Speculative additions anticipating future need are explicitly rejected. The exception is speculative-reserved entries (e.g., alternative-neutral Radix scales kept for the exploration page switcher), which require explicit listing in the appropriate registry plus a one-line justification per entry.

Applies to
primitive tokens semantic tokens components skills pages principles
Rule
Every shipped declaration has a current consumer in the brief that authorised it. No pre-emitted scales, no orphan tokens, no premature codification of patterns not yet shipped. A new entry without a consumer fails review.
Exception
speculative-reserved explicit registry listing + one-line justification per entry. Today: RESERVED_COLOR_SCALES + RESERVED_TOKENS in pipeline/tokens-parser.test.js (e.g., the 6 alternative-neutral scales for the data-neutral exploration switcher).
Mechanism · CI
tokens-parser.test.js · test 19 of 19 primitive-layer guardrail · every --nuri-* must be consumed (direct var() or via an alias chain) or explicitly speculative-reserved
Mechanism · upstream
AGENTS.md hard-rule 20 working-session anti-goals per-skill anti-goals spec-authoring agents see the rule before they write — cascade across AGENTS.md, the working-session template, and the 5 skill files that author shippable surfaces.
Reason
Audit cost grows linearly with speculative accumulation. The n=5 evidence trigger (shadow tokens · orphan font-size primitives · absolute line-height primitives · status placeholders · premature shadow atomic-pattern codification) all accumulated across earlier sessions and were only caught by manual audit at N+5.7. The principle closes that gap upstream so the next spec-authoring agent doesn't reproduce the drift class.
See
decisionlog.md decision 30 · roadmap/N+5.7.md (cleanup evidence) · roadmap/N+5.7.1.md (this lock) · RISKS.md R3 (pipeline-resolution divergence)
