Live scope

Switch theme / accent / neutral / font on the toolbar. Every control writes a data-* attribute on the scope wrapping this card; the nested <nuri-button>s re-resolve their semantic + component tokens through it — no JS re-render, just the cascade.

API

Five dimension attributes (all optional) plus an optional preset name. The web ↔ RN prop set is 1:1 minus font — every dimension below is a field on NuriThemeContext on RN except font, which is a web-only emulation overlay (amendment 27.1). Only mode is renamed across the boundary: the attribute is data-theme (the CSS files have used it since v0), the public prop is mode.

Prop data-* attribute Type RN Description
mode data-theme "light" | "dark" mode field The only renamed dimension (prop mode → attribute data-theme). Default light.
accent data-accent "lilac" | "neutral" | … accent field The active accent for the subtree. Default lilac (decision 8).
neutral data-neutral "cream" | "gray" | … neutral field The neutral colour scale. Default cream at build (decision 31).
font data-font "ios" | "android" | "pixel" — web-only Emulation overlay so designers preview platform system fonts in the docs. Not migrated — RN inherits the platform's system stack natively (amendment 27.1).
density data-density future density field Reserved. Auto-discovered by the emitter once its CSS cascade blocks land (decision 28).
name (preset) string — web-only Optional registry preset (window.NuriScopes.register(name, dims)). Direct props always win over a preset's dimensions.

Web mechanism · the cascade

On the web, <nuri-scope> is a thin custom element. It mirrors each dimension prop onto the matching data-* attribute on its own element, then gets out of the way with display: contents. There is no shadow DOM, no style injection, no re-render — the CSS cascade does the rest.

Mirror
prop → data-* on the element. mode→data-theme, every other dimension 1:1 (accent→data-accent, …).
Re-resolution
Semantic tokens (and component tokens) are re-declared on every scope-providing selector — :root, [data-accent], [data-theme] — so their var() references re-resolve at the new scope instead of inheriting an already-resolved literal (decision 9).
Invisible
display: contents the element contributes no box; children lay out exactly as if the wrapper weren't there.
Why "scope", not "theme"
Unistyles uses "theme" for a complete, globally-registered token configuration. <nuri-scope> is a different thing — a multi-dimensional cascade scope. The distinct word avoids confusion when devs read both libraries.

The element, in full:

<!-- ambient: whatever <html data-*> says -->
<nuri-button variant="solid">Inherits ambient accent</nuri-button>

<!-- a local scope: neutral accent, dark mode, for this subtree only -->
<nuri-scope accent="neutral" mode="dark">
  <nuri-button variant="solid">Re-resolves through the scope</nuri-button>
</nuri-scope>

Both the page-level <html data-accent> (Tier 1) and a leaf component's own accent prop (Tier 2) are the same mechanism at different DOM depths. <nuri-scope> is the generic, content-free version you reach for when you want to re-scope an arbitrary subtree.

RN counterpart · the context

RN has no CSS cascade, so the subtree-scope semantic is expressed with a single React Context. The shape is orthogonal, not a cross-product theme registry: pre-computing every (∏ dimensions) tuple scales O(∏ dims) — 8 accent × 3 font × 2 density × 7 neutral = 336 themes at the roadmap dimensions — and would break the composability that decisions 1, 6, 9 and P5 are predicated on (decision 27).

Single Context
NuriThemeContext carries one entry per live dimension. Today that is mode + accent only; density / neutral are reserved but not yet context entries until their web tokens exist (P11). Defaults mirror the web <html data-*> defaults: mode: 'light', accent: 'lilac'.
Merge-on-override
<NuriScope accent="neutral"> reads the ambient value and emits { ...ambient, accent: 'neutral' }. Unspecified dimensions inherit; specified ones win. No dimension is "all-or-nothing".
Props 1:1 (minus font)
mode · accent · density · neutral — same names, same value sets as the web attributes. font is web-only emulation overlay; no Font type ships (amendment 27.1).
One useContext
Component-side consumption is a single const { accent, mode, … } = useContext(NuriThemeContext), then destructure. No per-dimension Provider chain.
Unistyles posture
Consumed for stylesheet ergonomics only, if at all. Its theme registry is bypassed — runtime values come from NuriThemeContext, not UnistylesRuntime.setTheme(…) (which would have to enumerate ∏ dims tuples).

Shipped shape (N+13 · the working code now in _shared.tsx, tsc-confirmed). The type carries only the two live dimensions; density / neutral join it when their web tokens land:

type NuriThemeValue = {
  mode: Theme;
  accent: Accent;
  // density / neutral RESERVED · not entries until their web tokens
  // exist (P11). No `font` — web-only per amendment 27.1.
};

const NuriThemeContext = React.createContext<NuriThemeValue>({
  mode: 'light',
  accent: 'lilac',
});

const NuriScope: React.FC<Partial<NuriThemeValue> & { children: React.ReactNode }> = ({
  children,
  ...overrides
}) => {
  const ambient = React.useContext(NuriThemeContext);
  return (
    <NuriThemeContext.Provider value={{ ...ambient, ...overrides }}>
      {children}
    </NuriThemeContext.Provider>
  );
};

Cascade ↔ Context · mechanism delta

The prop API is identical; the runtime that resolves it is not. The web uses the CSS cascade (declarative, re-resolves on attribute change); RN uses a single Context value merged down the tree. Same intent — re-scope an arbitrary subtree — different machinery.

Aspect Web (<nuri-scope>) RN (<NuriScope>)
Carrier data-* attributes on the element Fields on a single NuriThemeContext value
Resolution CSS cascade re-resolves var() chains at the scope (decision 9) resolveToken(tokens, path) dereferences the live (accent × theme) slice at render
Override A nested scope re-declares only the attributes it sets; the rest inherit via the cascade Merge-on-override: { ...ambient, ...overrides } — unspecified dimensions inherit
Layout footprint display: contents — zero box Context Provider — zero box (no View wrapper needed)
font dimension Supported (emulation overlay) Absent — platform system stack inherited natively (27.1)
Reactivity Attribute change → cascade re-resolves, no JS Context value change → React re-renders consumers

Dimensions

Five orthogonal dimensions. Four migrate to NuriThemeContext as fields; font is web-only. The dimensions never combine into a pre-computed theme tuple — each is set, inherited, and overridden independently (decision 27).

Behavioural delta · web ↔ RN

Props are 1:1 (minus font), but the runtime mechanism differs — and the multi-dimensional case carries a cost the single-dimension matrix hasn't yet exercised. The friction register in RISKS.md R1 is the single source of truth; the rows below are this page's view of the two scope codes.

Code Web RN Delta the migration agent honours
F-SCOPE-1 One element re-scopes N dimensions at once via N data-* attributes One composite <NuriScope> Provider merges N dimensions into one Context value Use the composite, not one Provider per dimension (linear nesting).
