API

The props signature — a strict subset of IconButton's. There is no disabled, no label, no onPress: IconAvatar is decorative and aria-hidden, so it carries no accessible-name burden (F-ARIA-LABEL-1 does not apply · decision 50). web ↔ RN is 1:1 (see docs/migration-tests/button-matrix/index.tsx).

Prop Type Default Description
name string (registry key) required Kebab-case glyph name from the icons.js registry. Forwarded to the inner <nuri-icon size="md">.
variant "solid" | "soft" | "ghost" | "subtle" "soft" Surface treatment. solid fills with the active accent, soft uses neutral chrome, ghost is transparent. subtle is avatar-only: like ghost, but the glyph reads in border-strong for the lowest emphasis. solid/soft/ghost mirror IconButton's semantic surfaces (rest state only).
accent "lilac" | "neutral" inherited Self-scope override (Tier 2). Writes data-accent on the inner element. Only affects solid; soft, ghost and subtle ignore it.
fill boolean false Selects the filled glyph weight (passes through to the inner <nuri-icon fill>). Reads well for decorative, badge-like avatars.

Anatomy & parts

IconAvatar is a two-part component: the circular root surface owns size, shape, and the variant fill; the icon part is the wrapped <nuri-icon> glyph, inheriting colour via currentColor. Unlike IconButton, the root is a non-interactive <span> — no <button>, no focus, no role.

Part Element (web) Element (RN) Owns
root span View size (48px circle), shape, variant background
icon nuri-icon Icon the glyph (md weight · 28px), inherits root color

Variants

Four variants. solid, soft, and ghost are the SAME semantic surfaces IconButton uses (rest state only): solid carries accent identity; soft and ghost are chrome-only. subtle is avatar-only — like ghost, but the glyph reads in border-strong for the lowest-emphasis decorative treatment (no IconButton counterpart). Each shown with and without fill. Default is soft.

In context

IconAvatar's home is the leading slot of an activity / list row. The text label is the accessible content; the avatar is decorative (aria-hidden). Separators between rows are Separator.

Behavioural delta · web ↔ RN

IconAvatar is the rare near-zero-delta component: being decorative, it inherits none of IconButton's interaction deltas (F-PRESSED-1 / F-FOCUS-1 / F-DISABLED-1 / F-ARIA-LABEL-1 / F-TOUCH-TARGET-1 all N/A). The only cross-platform seam is the glyph render mechanism it shares with every Icon consumer — and that is now closed.

Friction Surface Web React Native Resolution
F-ICON-RN-1 glyph render Inline <svg> path keyed by name + weight (via <nuri-icon>) react-native-svg SvgXml over the shared typed registry (build/icons.ts) Closed in N+6.8 (decision 48). IconAvatar is the first NEW consumer to ship against the resolved Icon — composes the real RN Icon, no shim.
F-DECORATIVE-1 accessibility Host aria-hidden="true" · not focusable · no role accessibilityElementsHidden + importantForAccessibility="no-hide-descendants" Both platforms hide the avatar from AT entirely. The adjacent text label is the accessible content. No name to derive (contrast: IconButton's F-ARIA-LABEL-1).

Roadmap

  • planned
    Polymorphic <nuri-avatar> (image / initials)

    The bare Avatar name is deliberately reserved (decision 50) for a future primitive that also renders a photo or initials, not just an icon. IconAvatar is the icon-only leaf; promoting to a polymorphic Avatar is deferred until a real image/initials consumer surfaces (P11 · don't design speculatively).

  • planned
    Size scale

    N+6.9 locks a single 48px size (the same geometry as IconButton · decision 50). A multi-size matrix (e.g. list-leading vs header-hero) arrives only when two real sizes are needed — until then, one size keeps the surface minimal.

  • planned
    RN spec long-term home

    Like the other primitives, the IconAvatar RN definition lives in the migration-test pair pending a long-term home. Re-evaluated across the whole set, not per-component.
