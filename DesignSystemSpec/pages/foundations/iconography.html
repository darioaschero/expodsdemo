Approach

Icons come from Phosphor (@phosphor-icons/core) — a consistent, open-licensed set drawn on a 256×256 grid. Nuri ships a hand-curated 17-glyph subset, not the full library: each glyph is stored as inline <path> markup in icons.js, three weights apiece (regular · bold · fill). Catalog expansion is deliberate and governed — new glyphs join through a separate session, never ad hoc.

Two sizes, each coupled to a weight. Rather than expose an open weight prop, Nuri couples weight to size: md renders the regular weight, sm renders bold (a heavier stroke holds up at the smaller box). Setting fill overrides both and renders the solid fill weight. This is an opinionated reduction — fewer knobs, consistent optical weight across a screen.

Colour inherits via currentColor. <nuri-icon> has no tone, accent, or color prop — the rendered SVG paints with fill="currentColor", so it takes the text colour of whatever consumes it. A button, a list item, or a status surface controls the icon's colour by setting its own text colour; the icon never decides.

The box dimensions are dispatched from the existing semantic --nuri-size-* vocabulary — md → size-sm (28px), sm → size-xs (18px). No new --nuri-icon-size-* family was minted; Icon reuses two leaves of an existing scale via attribute dispatch, the same shape Stack/Box use for their props (decision 37).

API

The props signature. web ↔ RN is 1:1 at the props layer — every prop below exists on both the <nuri-icon> custom element and (when implemented) the RN <Icon> component, with the same name and accepted values. The RN renderer is deferred (see Roadmap).

Prop Type Default Description
name string (registry key) required Kebab-case glyph name from the icons.js registry. Unknown names render nothing and warn [NuriIcon] unknown name "…".
size "md" | "sm" "md" Box size + coupled weight. md → 28px regular; sm → 18px bold.
fill boolean false Presence forces the solid fill weight, overriding the size-coupled weight.

Examples:

<nuri-icon name="vault"></nuri-icon>            <!-- md · regular -->
<nuri-icon name="vault" size="sm"></nuri-icon>  <!-- sm · bold -->
<nuri-icon name="vault" fill></nuri-icon>       <!-- md · fill -->
<nuri-icon name="warning" size="sm" fill></nuri-icon>  <!-- sm · fill -->
Weight is not a prop There is no weight attribute. Weight is coupled to size + fill by design (decision 38). Raw-weight access is a deferred Roadmap item.

Token mapping

size prop → semantic size leaf → CSS var. Icon mints no component tokens — the size prop dispatches directly onto two leaves of the semantic --nuri-size-* scale via attribute selector (decision 37 enforcement · empty @layer tokens, no build/components/icon.ts).

size weight (default) box dimension semantic leaf value
md regular width / height --nuri-size-sm 28px
sm bold width / height --nuri-size-xs 18px

Roadmap

  • planned
    Raw-weight prop

    Weight is coupled to size today. A future weight prop could unlock the raw Phosphor weights (thin / light / duotone) — gated on a real consumer that needs to break the coupling. Unlock criterion is an Open question.

  • planned
    Brand glyphs

    Nuri-specific marks (logo, token glyphs) that aren't in Phosphor. Would extend the registry shape with a separate source provenance.

  • planned
    Pipeline auto-import

    Today the registry is regenerated by hand (throwaway extraction script). When the catalog batches its next expansion, promote extraction into the pipeline/ workstream so icons.js regenerates mechanically from a name list.

  • planned
    RN renderer

    The web side ships inline SVG. The RN consumer provides the actual renderer (react-native-svg or similar) against the same registry + props. Spec-only today; the RN home is an Open question (target N+6.4 when IconButton lands as the first consumer).
