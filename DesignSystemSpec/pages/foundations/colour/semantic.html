How to read these tables

Set · core

Each row is one semantic token, expanded across both modes structurally. Reference Light and Reference Dark are the primitive var(...) declarations the cascade lands on under data-theme="light" and data-theme="dark" respectively — the reference itself can change per theme (e.g. block 1 declares --nuri-bg-canvas as neutral-1-light; block 2 redeclares it as neutral-1-dark). Light and Dark are the resolved hex values from those walks. The chip inside each Reference cell renders the same colour the chip's cascade context produces — light chip on Reference Light, dark chip on Reference Dark — so you can read the matrix at a glance without toggling anything.

Chrome tokens (bg-*, text-*, border-*, focus-ring) depend only on data-theme. Accent tokens depend on data-theme × data-accent; the two accent sections below — Accent · neutral and Accent · lilac — split the accent dimension structurally so each token's value resolves once per accent. Read tokens-semantic.css for the cascade ordering of the six blocks that produce these values; per decision 33, each canonical block carries an inline Format B comment showing the same matrix the tables here render.

Focus

Accent · neutral

Saturated-scale accent. accent-fg and accent-bg-subtle* are same-scale across themes; accent-solid, accent-solid-pressed, and accent-on-solid are INVERSE — the light value comes from the dark side of the neutral primitive scale and vice versa. The matrix below renders both modes side-by-side; the same matrix is documented inline at block 1 of tokens-semantic.css per decision 33.

Accent · lilac

Bright-scale accent. accent-fg and accent-bg-subtle* adapt per theme (block 6 redeclares); accent-solid, accent-solid-pressed, and accent-on-solid are FROZEN per P4 — block 6 intentionally omits them, so the same lilac value resolves in both Light and Dark cells. The matrix is documented inline at block 5 of tokens-semantic.css.
