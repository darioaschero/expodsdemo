Approach

Sizing is the vocabulary min-height and width consume. Roles read as "xl touch target" rather than "60 pixels" — the name carries intent (this is a primary action button), the value can evolve without rewriting consumers.

The seven leaves are a T-shirt scale (xs · sm · md · lg · xl · 2xl · 3xl) that aliases the --nuri-px-N primitive layer (decision 32). Sizing is asymmetric vs Spacing by design: it anchors larger (touch targets, control heights, 18–90 px); spacing anchors smaller (between-element values 2–36 px). Same T-shirt vocabulary, different range.

--nuri-button-min-height is the first live consumer: it aliases var(--nuri-size-xl) = 60 px (comfortable primary action). Solid + soft buttons both inherit it.

Cascade-invariant today — the value of --nuri-size-xl is identical across every (theme × accent) combination, so the table below renders single-Value. The dependency-driven trigger from decision 33 promotes the row to Format B automatically if a future dimension (density · platform) splits the resolution.

Tokens

Set · core

Each row is one semantic sizing leaf. Reference is the primitive it aliases (--nuri-px-N · N = literal pixel value · decision 32); Value is the resolved CSS literal that the RN pipeline emits as a JS number into build/tokens.ts under the size namespace.

Roadmap

  • planned
    More component consumers

    Today only --nuri-button-min-height consumes a sizing leaf (size-xl). IconButton / Tag / SegmentedControl land in N+6.3 + later; each will alias a sizing leaf inside its own @layer tokens block.

  • planned
    Density dimension

    If a future data-density="compact" block lands, the same seven leaves can re-map to a tighter set of primitives. The cascade-classifier auto-promotes the rows to Format B once the second cascade context exists.
